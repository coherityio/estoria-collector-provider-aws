Proposed AWS Collectors for estoria-collector-provider-aws
=========================================================

Design assumption: one Collector class per entity type exposed by the AWS
provider. Naming convention (suggested):

    Aws<Vpc>Collector for entity type "Vpc"
    Aws<Subnet>Collector for entity type "Subnet"

Below is an extensive (though still finite) list of entity types grouped by
AWS domain. Each bullet corresponds to one potential Collector implementation.

[High-Level Notes]
---------------------------------------------------------------------
- This list is intentionally broad; you would prioritize implementation
  based on use cases (e.g., security posture, cost, inventory, topology).
- Some entity types may be modeled as attributes of another collector
  instead of separate collectors, depending on how you design
  `CloudEntity` and relationships (e.g., DynamoDB indexes, S3 bucket
  ACLs, etc.).
- For large/streaming datasets (e.g., individual CloudTrail events,
  X-Ray traces, or log records), you might avoid per-event collectors
  and instead model higher-level configuration or aggregated findings.
- Cross-cutting concepts like tags can be exposed as attributes on
  each resource entity instead of a separate `TagCollector` unless a
  distinct entity model is required.
